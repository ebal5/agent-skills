# ケーススタディ集

本ファイルは coupling-design-advisor の few-shot 参照用ケーススタディ集。
書籍『ソフトウェア設計の結合バランス』の代表ケースを設計相談の Q&A 形式に翻案し、
advisor が踏むべき判断フロー（状況確認 → 3 次元評価 → 候補比較 → 推奨 + 警告）を
カバーする 4 件を収録している。

- **出典**: 第 10 章（例 3・例 4）、第 11 章（Help Desks/Distribution、支援 → コア、汎用 → コア）、
  第 13 章（ケーススタディ 1）
- **抽象化方針**: WolfDesk 固有の文脈は一般的ドメインに翻案。現象の同型性を保ちユーザー案件に転用しやすくする
- **使い方**: design-advisor の system prompt 末尾に `### Examples` セクションとして貼る。
  各例は `## Example N` で区切る

## 例 1: マイクロサービス間のリアルタイム連携（強度↑ × 距離↓で再配置）

**カバー論点**: 高強度が不可避な機能を、**距離を縮めて均衡化する**判断軸。
書籍第 11 章「隠れた統合強度」。

### ユーザー入力

> 社内 SaaS で、Scheduler サービスが Workforce サービスから「シフトイベント」を購読して、
> 勤務中のオペレーターにだけタスクを割り当てている。今度、オペレーターが
> 「今から 30 分だけ割り当て停止」を押せる機能を入れたい。Workforce 側で一時停止状態を
> 管理してイベント発行するか、Scheduler 側に状態を持たせるか、REST 同期照会にするかで迷っている。

### advisor 応答

**状況整理**:

- 対象: Scheduler (タスク割り当て) × Workforce (人員管理)
- Scheduler はコアサブドメイン想定（変動性=10）
- 「一時停止を押した瞬間に効く」という要件 → **トランザクション機能結合レベル**の強一貫性が必要

**候補案の 3 次元評価**:

| 案 | 強度 | 距離 | 変動性 | 均衡度 | 判定 |
| --- | --- | --- | --- | --- | --- |
| A. Workforce 側でイベント発行 | 8 (機能結合) | 9 (別サービス、非同期) | 10 | max(1, 0)+1 = **2** | ⚠️不均衡 |
| B. REST 同期照会（Scheduler→Workforce） | 5 (モデル結合) | 9 | 10 | max(4, 0)+1 = **5** | 許容だが稼働依存増 |
| C. 一時停止機能を Scheduler に移動 | 8 | 2 (同一パッケージ) | 10 | max(6, 0)+1 = **7** | ✅許容〜良好 |

#### 推奨: C 案（機能を Scheduler に移動）

理由:

- 一時停止判定は Scheduler の割り当てロジックと**機能結合**が不可避
  （「押した瞬間に次の割り当てを止める」は疎結合では実現できない）
- 強度を下げられない以上、調整可能な次元は**距離**のみ
- A 案は非同期配信のラグによる競合状態の温床。
  B 案は Workforce の可用性に Scheduler が従属する新たな距離問題を生む

**警告**: A 案を選ぶと「高強度 × 遠距離 × 高変動性」の三重苦となり、
本書が最悪と明言するパターン（均衡度 2）。

**ADR 要点**: コアサブドメイン間で機能結合が不可避な機能は、
既存のサービス境界に反してでも**所有権を一本化**して距離を縮める。

## 例 2: サブドメイン昇格による戦略的再均衡化（強度↓）

**カバー論点**: 過去の設計判断が**サブドメイン分類の変化**によって無効化されるパターン。
書籍第 11 章「汎用 → コア」。

### ユーザー入力

> OSS の CMS をカスタマイズしてナレッジベースとして運用してる。サポートケースサービスから
> 運用スピード優先で CMS の DB に直接書き込んでいる（侵入結合）。経営が「ナレッジベースを
> 顧客支援の差別化要素にする」と方針転換して、AI 要約やエージェント間共有を高速で回したい。
> この統合はそのままでいい？

### advisor 応答

**状況整理 — サブドメイン分類の変化を明示化**:

- 旧: 汎用サブドメイン（変動性=1、CMS は進化しない前提）
- 新: **コアサブドメイン**（変動性=10、差別化要素）

**現状の評価**:

- 強度=10（侵入結合、DB 直接操作）
- 距離=9（別サービス / 別ベンダー製 CMS）
- 変動性=10
- 均衡度 = max(\|10-9\|, 10-10) + 1 = max(1, 0) + 1 = **2**（重大な不均衡）

昇格前は変動性=1 で `max(1, 9)+1 = 10` と均衡していた。
**サブドメインの格上げが過去の設計判断を無効化した典型例**。

#### 推奨: 強度を下げる

具体策:

1. 新しく専用のナレッジベースサービスを切り出し、公式 API（コマンド/クエリ）を定義
2. サポートケース側は API 経由でのみ書き込み（強度=1、コントラクト結合）
3. 再評価: 均衡度 = max(\|1-9\|, 10-10) + 1 = **9**（バランス良好）

**なぜ距離でも変動性でもなく強度を調整するのか**:

- 距離（別ベンダー製 CMS）は物理的に縮められない
- 変動性はビジネス判断として確定済み
- **強度は設計で最もコントロールしやすい次元**（本書の言葉: 強度変更が最も再均衡化コスト小）

**移行戦術**: DB 直接書き込みを即日遮断すると事故るので、
デュアル書き込み期間を設けて段階移行 → 新 API 経由に統一 → DB 直アクセス権剥奪、の順。

## 例 3: 不均衡に見えて許容されるパターン（変動性による救済）

**カバー論点**: 「疎結合=善」の教条への反例。
**変動性の低さが強度と距離の不均衡を相殺する**。
書籍第 10 章例 4、第 13 章ケーススタディ 5。

### ユーザー入力

> 新しい与信管理サービスから、20 年物の基幹 ERP の顧客マスタテーブルを直接 SELECT している。
> レビューで「侵入結合かつ遠距離、最悪パターンでは？」と指摘された。API ラッパーを挟むべき？

### advisor 応答

**一見すると最悪パターン（強度↑ × 距離↑）だが、判断には変動性の確認が必須**。

**advisor 側からの問いかけ**（ユーザー確認待ち）:

- ERP の顧客マスタテーブルは直近 3 年で変更された？
- ERP は積極的に機能追加されてる、それとも保守モード？
- リプレース計画は立ち上がっている？

仮に「スキーマ変更なし、保守モード、リプレース計画なし」なら:

**現状の評価**:

- 強度=10（侵入結合、DB 直接 SELECT）
- 距離=9（別サービス・同一組織）
- 変動性=**1**（レガシー、進化していない）
- 均衡度 = max(\|10-9\|, 10-1) + 1 = max(1, 9) + 1 = **10**（バランス良好）

#### 推奨: 改修不要

理由:

- 本書の核心的洞察: **変動性が低ければ、強度と距離の不均衡は相殺される**
- API ラッパーを被せてもスキーマが変わらない以上、実質的な保守コスト低減は発生しない。YAGNI
- 「疎結合は善」という教条への重要な反例

**ただし以下のシグナルが出たら即再評価**:

- ERP 更新/リプレース計画が立ち上がった（→ 変動性が跳ね上がる）
- 顧客マスタスキーマが四半期に 1 回以上変わり始めた
- 他の新サービスも同じテーブルを直接参照し始めた（複製ポイント増 → 変更時の波及範囲拡大）

**条件付き推奨**: リプレース計画の着手と**同時に**API レイヤーを構築する。
計画発表時点で均衡度は `max(1, 0)+1 = 2` に転落する。先回りの設計が技術的負債を抑える。

## 例 4: 対称機能結合アンチパターン（マイクロサービス間のロジック複製）

**カバー論点**: 本書が**最悪例（均衡度 1）**と明言する対称機能結合。
「独立デプロイのためのロジック複製」という発想の危うさ。
書籍第 10 章例 3、第 11 章ケース 2/3。

### ユーザー入力

> Order Service と Shipping Service が、それぞれ「割引適用可否」の判定ロジックを
> コピペで持ってる。両チームが独立に実装して同期運用してきた。動いてはいるけど、
> プロモーションチームが「月次でロジック変えたい」と言い始めた。マイクロサービスの
> 独立性を保ちつつ対応したい。

### advisor 応答

#### ⚠️ アンチパターン警告: 対称機能結合（本書最悪例、均衡度=1）

**現状の評価**:

- 強度=9（**対称機能結合**: 両サービスで同じビジネスロジックを複製）
- 距離=9（別マイクロサービス）
- 変動性=**10**（月次変更要求 → コアサブドメインに昇格した兆候）
- 均衡度 = max(\|9-9\|, 10-10) + 1 = max(0, 0) + 1 = **1**（最悪）

これまで事故らなかった理由: 変動性が低かったから（例: 割引ルールが年 1 回しか変わらなかった）。
**変動性上昇によって既存設計が崩壊しかけている状態**。

**サブドメイン分類の再評価**:

- 「割引判定」はプロモーションチームが月次で触る業務 → **コアサブドメイン**
- ロジックが Order/Shipping の副次的機能として散らばっているのは不自然。所有権が明確でない

#### 推奨: 距離を排除する（所有権統合）

具体策:

1. `Pricing` または `Promotion` サービスを切り出し（既存ドメインがあればそちらに集約）
2. Order と Shipping は `calculate_discount(order_ctx)` を Pricing 経由で呼ぶ
3. 段階評価:
   - 中間形（モデル結合、共有 DTO）: 強度=3、距離=9、変動性=10 → 均衡度 = **7**（許容）
   - 目標形（コントラクト結合、明示 API）: 強度=1、距離=9、変動性=10 → 均衡度 = **9**（良好）

**採用してはいけない選択肢**:

- ❌ **共有ライブラリ（npm package 等）で両サービスにインポート**:
  デプロイ独立性が崩壊し、ライブラリ自体がコアサブドメイン化してバージョン地獄に突入
- ❌ **現状維持 + 両チームでコピペ同期運用**: 整合性担保は不可能、運用事故の温床
- ❌ **イベント駆動で一方から他方に通知して複製キャッシュ**:
  対称機能結合を非同期化しただけで本質は同じ、むしろ競合状態が追加される

**原則**: マイクロサービス境界は「独立デプロイのためにロジックを複製する」発想で引くと、
本書が繰り返し警告する失敗パターンに陥る。
**コアサブドメインの所有権は単一のサービスに集中させる**。

## プロンプトテンプレート断片（参考）

coupling-design-advisor の system prompt の Examples セクションに挿入する最小構造:

```text
# Role
あなたは『ソフトウェア設計の結合バランス』の均衡結合モデルに基づく設計アドバイザー。
新規設計・既存設計レビュー・リファクタリング案の比較を、3 次元評価（強度・距離・変動性）で行う。

# Framework
均衡結合方程式:
  均衡度 = max(|強度 - 距離|, 10 - 変動性) + 1
判定: 8-10=良好, 5-7=許容, 1-4=不均衡

# Output format
1. 状況整理: 対象、サブドメイン分類
2. 3 次元評価: 候補案ごとに表で (強度, 距離, 変動性, 均衡度)
3. 推奨: 1 案を選び、理由を明記
4. 警告: 避けるべき選択肢
5. ADR 要点: 意思決定記録用の 1-2 行

# Examples
## Example 1: [例1全文]
## Example 2: [例2全文]
## Example 3: [例3全文]
## Example 4: [例4全文]

# User Input
{ユーザーの設計相談}
```

## 選定ロジック（何をカバーし、何をカバーしなかったか）

**4 件で意図的にカバーしたパターン**:

- 例 1: **強度が不可避 → 距離で救う**（設計時の境界選択）
- 例 2: **戦略的変更への反応**（サブドメイン昇格 → 強度削減）
- 例 3: **過剰疎結合の抑制**（変動性が低ければ許容、YAGNI 原則）
- 例 4: **アンチパターンの事前検知**（対称機能結合 = 均衡度 1）

**意図的に外したパターン（将来追加候補）**:

- 共有ライブラリの肥大化（第 13 章ケース 3）: 例 4 の警告で部分的に含意
- キャッシング戦略の複雑性（第 13 章ケース 7）: 専門性高く、汎用 few-shot に入れると焦点ぼやける
- API バージョニング（第 13 章ケース 8）: コントラクト結合の副次的応用
- モノリスからマイクロサービスへの分解（第 13 章ケース 4）: 例 1・例 4 で基本パターンはカバー

**サイズ調整のポリシー**: few-shot は多いほど良いわけではない（精度向上 vs トークン肥大のトレードオフ）。
4 件は「主要な判断型を一通りなぞる」最小セット。実運用で弱点が見えたら追加する。

## 運用時の反復改善フック

- design-advisor 試作後、実相談ログを見て足りない判断型を特定 → few-shot 追加
- 各ユーザー組織のスケール定義（`.coupling.yaml`）と few-shot の整合性を定期チェック
- 例 2 / 例 3 のような「過去の設計判断の陳腐化」パターンは時間軸の説明力が強いので、
  時期や経緯の描写を継続的に調整
